## Bugs diagnosed + fixed

### 1. Chain silently 401'd on next-hop fetch — Vercel Deployment Protection

**Symptom:** Trip flipped to `status='ingesting'`, then nothing. All 3 uploads stayed `pending`. Zero `ai_runs`, zero `upload_chunks`.

**Diagnosis path:** Vercel log for the entry route `POST /api/ingest/[tripId]` showed the 2 external calls:

```
PATCH ...supabase.co/rest/v1/trips → 204 388ms      (status flip: ingesting)
POST  .../api/ingest/.../extract   → 401  99ms      (⚠️ next-hop rejected)
```

**Root cause:** Vercel Deployment Protection (Vercel Authentication) was set to "All Deployments", including production. `req.url` inside the function resolves to the deployment-hash alias (`eastern-314wlp836-...vercel.app`), which sits behind the SSO wall — only the canonical prod domain (`eastern-exe.vercel.app`) is exempt. The entry route's `waitUntil(fetch(nextHop))` hit the auth wall and died. No error log because `chain.ts` swallows fetch failures into `console.error` only.

**Fix:** User changed Project Settings → Deployment Protection → Vercel Authentication from "All Deployments" to "Only Preview Deployments". Production deployments are now public (as they should be for a user-facing app); preview branches stay protected.

No code change needed for this one.

### 2. WhatsApp zip `status='processing'` forever — 60s cap hit mid-media-upload

**Symptom (after fix #1):** Chain now fired. WhatsApp zip upload flipped to `processing`, sat there > 2 min. Two notes stayed `pending` (loop never reached them). No error in `ai_runs`.

**Root cause:** `extractFromUpload` parses the zip, then loops the `mediaFiles` array and uploads each image to storage sequentially (`src/lib/ingest/pipeline.ts:137`). Each upload is a network round-trip; a typical WhatsApp chat has dozens to hundreds of media files. Loop blows past the 60s Hobby cap, Vercel freezes the instance mid-`await`, the outer `try/catch` that would have flipped the upload to `failed` never runs — so the row is pinned at `processing` forever.

Same silent-death pattern as the places bug from the 0206 session, just earlier in the pipeline.

**Key insight:** A grep of the codebase showed media uploads are **never consumed downstream**. `participant_profiles` only reads `kind='other'` (notes). `trip_memory` / `places` only read `upload_chunks` (text). The image uploads were pure write-and-forget — deadweight blowing the budget.

**Fix landed:** Dropped the media upload loop entirely. Kept the zip text parse (that's what feeds LLM context).

```diff
  if (upload.kind === "whatsapp_zip") {
    const parsed = await parseWhatsAppZip(buf);
-   const mediaUploads: {...}[] = [];
-
-   for (const media of parsed.mediaFiles) {
-     // ... sequential storage.upload for every image in the zip
-   }
-
    return {
      extracted: [{ uploadId: upload.id, ..., text: parsed.text }],
-     mediaUploads,
+     mediaUploads: [],
    };
  }
```

Left the `mediaUploads` return field in place (the caller `runExtract` still spreads it into the upload-insert loop at `pipeline.ts:221`; empty array is a no-op). Minimal diff.
