## Context

Immediate follow-on from the 0259 session. After the WhatsApp media-upload loop was removed, a fresh test trip ingested the zip successfully — but the first person's notes got pinned at `processing` and the second person's notes stayed `pending`. Same silent-kill pattern as before, one step shallower in the pipeline: the now-faster-but-still-heavy zip parse was eating most of one shared 60s `/extract` budget, and the .txt notes right after it ran out of runway.

## Bug diagnosed

**Shared 60s budget across all uploads in `/extract`.**

`runExtract` loaded every pending upload for the trip and looped them inside one serverless invocation. Flow per iteration:

1. `setUploadStatus(id, 'processing')` ← flipped immediately
2. `downloadUpload` → `extractFromUpload` → `chunkText`
3. `setUploadStatus(id, 'processed')`

A real WhatsApp zip's download + parse chewed through the bulk of the 60s. When the loop moved to notes #1, step 1 ran (hence `processing`), step 2 hit the cap mid-`await`, Vercel froze the instance, the `catch` that would have flipped the row to `failed` never executed. Notes #2 stayed `pending` because the loop never reached it.

This is the same root cause as the earlier places-step bug — one Vercel 60s budget shared across work that can't all fit inside it.

## Fix landed — one-upload-per-hop

Split `/extract` into self-chaining per-upload invocations, same shape as `/profiles`.

### Pipeline changes (`src/lib/ingest/pipeline.ts`)

- **`listPendingUploadIds(tripId)`** (new) — returns ids ordered by `created_at` ASC. Filters on `status IN ('pending', 'processing')` rather than just `pending`. The `processing` inclusion recovers rows that got killed mid-await by a prior invocation — without it, a single timeout would pin a row out of reach of any retry.
- **`runExtractOne(tripId, uploadId)`** (new) — downloads, parses, chunks, inserts chunks for one upload in isolation. Flips status to `processed` on success, `failed` with message on caught error. Self-contained so the whole thing fits comfortably inside 60s even for big zips.
- **`tripHasChunks(tripId)`** (new) — `count({head:true})` on `upload_chunks`. Decides whether to chain to `/profiles` or skip to `/finalize` once the extract queue drains.
- **`runExtract`** — kept as a thin loop wrapper over `listPendingUploadIds` + `runExtractOne`. Still used by `runIngestion` (the local-dev seed path via `seed-test-trip` where the 60s cap is irrelevant).

### Route changes (`src/app/api/ingest/[tripId]/extract/route.ts`)

Rewrote as a self-chaining head-picker:

```
POST /api/ingest/{tripId}/extract
  → listPendingUploadIds() → []        → /profiles?i=0 or /finalize
  → listPendingUploadIds() → [id, ...] → runExtractOne(id); /extract (loop)
```

Deliberately no `?i=N` index. `listPendingUploadIds` filters on status, so the head of the returned list is always the next upload that still needs work. That avoids shrinking-list drift (if I'd used an index, `idx=1` on the second hop would point at the post-flip list where notes #2 sits at slot 0, skipping work) and makes the endpoint idempotent for manual retries.

## Expected effects

- Each upload gets its own 60s — a 40s zip no longer starves the 2s notes behind it.
- Total extract budget scales with N uploads × 60s (minus cold-start tax per hop, ~1–2s each).
- Stuck-at-`processing` rows self-heal: `listPendingUploadIds` will grab them on any subsequent `/extract` POST. Retry is as simple as firing that endpoint once.
- No change to the profiles/memory/places/finalize links in the chain — those were already fine.
