## Changes landed

### 1. Resume endpoint (`src/app/api/ingest/[tripId]/resume/route.ts`) — NEW

State-inspecting endpoint that determines which chain step to fire next. Idempotent — safe to call repeatedly. Logic:

```
POST /api/ingest/{tripId}/resume
  → uploads still pending/processing?    → /extract
  → no chunks (empty trip)?              → /finalize
  → zero profiles generated?             → /profiles?i=0
  → some profiles but not all?           → /profiles?i=N (resume index)
  → no trip memory?                      → /memory
  → no places?                           → /places
  → everything done but not finalized?   → /finalize
```

Returns `{ ok, resumed, reason }` as JSON with status 202. Uses `waitUntil(fetch(nextPath))` — same dispatch shape as the existing chain routes.

### 2. Auto-resume in `useTripStatus` (`src/hooks/useTripStatus.ts`) — MODIFIED

The client-side polling hook now detects when a trip has been stuck in `status='ingesting'` for more than 20 seconds and auto-fires `POST /api/ingest/[tripId]/resume`.

- Tracks ingestion start time via `useRef` (set when `trip.status === 'ingesting'` first observed)
- After 20s, fires the resume endpoint
- 30s cooldown between resume attempts to avoid spamming
- Resets timer when trip leaves `ingesting` (goes to `ready` or `error`)
- Runs as a separate `useEffect` alongside the existing realtime + polling subscriptions

### 3. Supabase server client `cache: "no-store"` — CONFIRMED IN PLACE

`src/lib/supabase/server.ts` already had the fix from the prior Claude session (wrapping `fetch` with `cache: "no-store"` to bypass Next.js 14's default fetch caching). No change needed — verified during review.

## Expected effects

- **No more manual `curl` kicks needed.** If a chain hop drops (the known failure mode), the client detects the stall after 20s and fires `/resume`, which inspects state and re-enters the pipeline at the correct step.
- **Self-healing for every observed stall scenario:**
  - `/extract` → `/profiles` drop: resume sees no profiles, fires `/profiles?i=0`
  - `/memory` → `/places` drop: resume sees no `ingest.places` in `ai_runs`, fires `/places`
  - `/places` → `/finalize` drop: resume sees all steps done but trip not `ready`, fires `/finalize`
  - Stuck `processing` uploads: resume sees pending uploads, re-enters `/extract` (which picks up stuck rows)
- **Tab must be open** for auto-resume to fire. If the user closes the browser during ingestion, the trip stays stuck until they revisit the URL (at which point the 20s timer starts and resume fires). For fully unattended recovery, a Vercel Cron sweep or queue (Inngest/QStash) would be needed.
