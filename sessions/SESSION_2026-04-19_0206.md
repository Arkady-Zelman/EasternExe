## Fixes landed â€” chained-endpoint ingestion

Split the monolithic `runIngestion` across five HTTP endpoints so each step gets its own 60s budget. Total budget: ~300s instead of 60s. No new infra required.

### New step functions (`src/lib/ingest/pipeline.ts`)

Refactored the old `runIngestion` into exported, independently-callable steps:

- `runExtract(tripId)` â€” Steps 1â€“3 (download uploads, extract text, persist chunks). Returns `{ hasUploads }` so callers can short-circuit.
- `listParticipantIds(tripId)` â€” helper for the profiles chain.
- `runProfileForParticipant(tripId, participantId)` â€” Step 4 for one participant (one LLM call per invocation).
- `runMemory(tripId)` â€” Step 5.

- `markIngestError(tripId, err)` â€” flips trip to `status='error'` when a chain link throws.

`runIngestion` still exists; it now just loops the step functions in-process (used by `seed-test-trip` for local-dev seeding where the 60s cap doesn't apply).

### Chain helper (`src/lib/ingest/chain.ts`)

```ts
chainStep(req, tripId, async () => {
  await runXxx(tripId);
  return `/api/ingest/${tripId}/nextStep`; // or null to end chain
});
```

Runs the step inside `waitUntil(...)`, fires the next-hop `fetch` from the step's returned path, marks the trip errored on any uncaught throw.

### New failure mode introduced

If a next-hop `fetch` itself gets dropped (Vercel instance restart, network blip), the chain stalls mid-pipeline and the trip sits at `ingesting`. Not mitigated. Options if it bites:

- Manual resume endpoint that inspects `ai_runs` + trip state and re-fires from the correct step.
- Cron sweep of trips stuck in `ingesting` > N minutes.
- Queue-based ingestion (Inngest / QStash / Supabase Edge Functions) â€” the proper long-term fix.
