## Bug diagnosed

**"London crew" trip on production stuck in `status='ingesting'` indefinitely.**

Verified via `/api/trips/<id>/snapshot`:
- `trips.status = 'ingesting'`, `trips.error = null`
- `destination_lat/lng = null` (Step 7 never ran)
- Zero rows in `ai_runs` for that trip (Steps 4‚Äì6 never ran)
- Zero uploads (user skipped bios + WhatsApp)

Root cause: **fire-and-forget async on Vercel serverless.** Both `/api/ingest/[tripId]/route.ts` and `/api/agent/route.ts` used:

```ts
void runIngestion(tripId).catch(...);
return NextResponse.json(...);
```

Vercel freezes the function instance the moment the response is sent. The first `await setTripStatus(tripId, "ingesting")` inside `runIngestion` gets through (which is why the UI flips to "ingesting"), but nothing after it runs ‚Äî including the outer `try/catch` that would have set the status to `error`.

Compounded by: user is on **Vercel Hobby**, so the declared `export const maxDuration = 300` was silently capped to 60s anyway.

## Fixes landed

### 1. Fire-and-forget ‚Üí `waitUntil`

Installed `@vercel/functions@^3.4.3`. Replaced `void foo().catch(...)` with `waitUntil(foo().catch(...))` in:

- `src/app/api/ingest/[tripId]/route.ts`
- `src/app/api/agent/route.ts`

Vercel now keeps the function instance alive until the promise resolves (up to `maxDuration`), instead of freezing on response.

### 2. `maxDuration` 300 ‚Üí 60

Same two routes. Matches the Hobby cap. The 300 was a lie ‚Äî explicit ceiling prevents future debugging confusion.

### 3. Empty-trip short-circuit in `runIngestion`

`src/lib/ingest/pipeline.ts` ‚Äî after loading `uploads`, bail early if `uploads.length === 0`:

- Geocode the destination (for the map center)
- Set `trips.status = 'ready'`
- Return

Saves ~20‚Äì40s of LLM calls on empty context (which would hallucinate garbage anyway) and fits comfortably inside Hobby's 60s on any trip.
